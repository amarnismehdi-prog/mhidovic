| **0.9.61-alpha** | 127 | **Settings icons — drop the style attribute entirely.** Even with `android:layout_width="28dp"`/`layout_height="28dp"` inlined on the View tag in 0.9.6, the leading icons still rendered as tiny white dots on the BYD device. Compared to the `+`/`−` slider icons (which render correctly), the only remaining difference was that the leading `<ImageView>` still had `style="@style/Widget.DashCast.ListItem.Leading"` applied. Removed the `style="…"` attribute from the 7 leading `<ImageView>` instances in `activity_settings.xml` and inlined every needed attribute directly on the View tag (`layout_width=28dp`, `layout_height=28dp`, `layout_marginEnd=16dp`, `layout_gravity=center_vertical`, `scaleType=fitCenter`, `tint=@color/md_on_surface_variant`, plus the existing `src`). No other change. |
| **0.9.6-alpha** | 126 | **Settings icons — the actual root cause.** Reported on the car after 0.9.5: the icons in `Comportement` and `À propos` were rendering as tiny white dots, regardless of the `android:tint` value. Root cause finally found: on this BYD Android 10 runtime, `android:layout_width` and `android:layout_height` declared inside a `<style>` (here `Widget.DashCast.ListItem.Leading` with `28dp × 28dp`) are not honored by `LayoutInflater` because layout params are extracted from the View tag itself, not from the applied style. The ImageView therefore fell back to a near-zero size, so the vector was rendered as a 1–2 px white dot. Fix: inlined `android:layout_width="28dp"`, `android:layout_height="28dp"`, `android:layout_marginEnd="16dp"` on every leading `<ImageView>` in `activity_settings.xml` (7 occurrences). The style still applies `layout_gravity`, `scaleType` and other non-layout attributes. No other change. |
| **0.9.5-alpha** | 125 | **Settings icons — third and final fix attempt.** In 0.9.3 the tint was on the vector drawable (didn't work). In 0.9.4 it was inlined as `android:tint="?attr/colorOnSurfaceVariant"` on each `<ImageView>` (still didn't work on the BYD device — the theme attribute reference returned a transparent value at runtime, so every leading icon in Comportement, À propos and the +/- slider icons rendered as empty space). Final fix: replaced every `android:tint="?attr/colorOnSurfaceVariant"` with `android:tint="@color/md_on_surface_variant"` (8 occurrences in `activity_settings.xml`) and `android:tint="?attr/colorOnPrimaryContainer"` with `android:tint="@color/md_on_primary_container"` on the Welcome hero `ic_cast`. The color resource is defined in both `values/colors.xml` (`#43474E`) and `values-night/colors.xml` (`#C3C6CF`), so DayNight is still respected through resource-qualifier resolution instead of theme-attribute indirection — which is what actually works on this Android 10 BYD runtime. No other change. |

The hero `ic_cast` in `activity_welcome.xml` got an inline `android:tint="?attr/colorOnPrimaryContainer"`. **Settings — `Vérifier` button now triggers the real OTA flow**: in 0.9.3 the `btn_check_update` opened the GitHub Releases page in a browser, which did not match the 3-dot-menu OTA.
